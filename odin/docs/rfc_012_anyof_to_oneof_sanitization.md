# RFC-012: `anyOf` Keyword Sanitization — Structural Conversion to `oneOf`

**Version:** 1.1  
**Author:** Chason Tang  
**Date:** 2026-02-27  
**Status:** Implemented

---

## 1. Summary

RFC-006 placed `anyOf` in the `ALLOWED_KEYWORDS` whitelist under the assumption of native Antigravity support (§4.3, rationale #8: "Native `allOf`/`anyOf`/`oneOf` Support"). Production validation has disproven this — Antigravity rejects requests containing `anyOf` with 400 errors. This RFC adds a **Phase 1c semantic transform** that converts `anyOf → oneOf` (a natively supported keyword) to the RFC-006 sanitizer pipeline, preserving full sub-schema structure without description field pollution.

**Semantic note:** `oneOf` (exclusive OR: exactly one branch matches) is **strictly more restrictive** than `anyOf` (inclusive OR: at least one branch matches). The two are not generally equivalent. However, for **type-union patterns** — the dominant `anyOf` use case in tool schemas, where branches are mutually exclusive by `type` — and for the purpose of **guiding model generation** (not validating ambiguous inputs), the behavioral difference is nil. See §4.3 rationale #2 for the formal analysis.

**Key decision:** Description-based semantic preservation (appending `anyOf` semantics to the `description` field) was evaluated and **rejected**. The `anyOf → oneOf` structural conversion preserves all sub-schema types, constraints, and hierarchy — making description hints both unnecessary and inferior. See §4.3 for the detailed three-approach comparison.

## 2. Motivation

`anyOf` is used in Anthropic tool `input_schema` definitions for type unions — the JSON Schema mechanism for expressing "this parameter accepts one of several types." Common patterns include:

- **Union types:** `anyOf: [{ type: "string" }, { type: "number" }]`
- **Nullable types:** `anyOf: [{ type: "string" }, { type: "null" }]` (generated by Python `Optional[str]`)

With `anyOf` in `ALLOWED_KEYWORDS`, these schemas pass through the sanitizer unchanged and reach Antigravity, which rejects them with opaque 400 errors — causing **complete request failures** for any tool definition containing `anyOf`.

**Production evidence** (from `odin/logs/body.json`):

```json
"contact": {
    "anyOf": [
        { "type": "string", "format": "email" },
        { "type": "number" }
    ]
}
```

This tool schema was forwarded to Antigravity unchanged and triggered a 400 rejection.

**Root cause:** RFC-006 §4.3 rationale #8 stated "Native `allOf`/`anyOf`/`oneOf` Support — These are natively supported by Antigravity." This was incorrect for `anyOf`. The keyword must be sanitized.

## 3. Goals and Non-Goals

**Goals:**

- Eliminate `anyOf`-caused 400 errors in production.
- Preserve full structural type information from `anyOf` schemas (all sub-schema types, constraints, and hierarchy).
- Maintain the single-stage whitelist architecture from RFC-006 with minimal additions.

**Non-Goals:**

- Full `anyOf` semantic evaluation (multi-schema overlap detection, schema intersection analysis) — `anyOf`/`oneOf` semantic divergence only affects validation of overlapping schemas, not model generation.
- Changes to `allOf` or `oneOf` handling — both remain in `ALLOWED_KEYWORDS`.
- Description field population for `anyOf` content — the structural conversion makes this unnecessary (see §4.3).

## 4. Design

### 4.1 Overview

A new Phase 1c semantic transform is added to the RFC-006 sanitizer pipeline:

```
┌─────────────────────────────────────────────────────────────┐
│                   Anthropic input_schema                    │
│                (JSON Schema 2020-12 full)                   │
└─────────────────┬───────────────────────────────────────────┘
                  │
    ┌─────────────▼──────────────────┐
    │  Phase 1: Semantic Transforms  │
    │  1a. const → enum              │
    │  1b. exclusiveBounds → min/max │
    │  1c. anyOf → oneOf        ← NEW│
    └─────────────┬──────────────────┘
                  │
    ┌─────────────▼──────────────────┐
    │  Phase 2: Whitelist Filter     │
    │  (anyOf removed from whitelist)│
    └─────────────┬──────────────────┘
                  │
    ┌─────────────▼──────────────────┐
    │    Antigravity-safe schema     │
    └────────────────────────────────┘
```

**Keyword registry changes:**

- `ALLOWED_KEYWORDS`: remove `'anyOf'` (23 → 22 keywords)
- `KNOWN_UNSUPPORTED_KEYWORDS`: add `'anyOf'` (38 → 39 keywords)

After Phase 1c converts all `anyOf` to `oneOf`, Phase 2's whitelist passes `oneOf` through (it remains in `ALLOWED_KEYWORDS`) and strips any unconverted `anyOf` (the collision edge case where both `anyOf` and `oneOf` coexist on one schema node).

### 4.2 Detailed Design

#### 4.2.1 Phase 1c: Convert anyOf → oneOf

```javascript
/**
 * Phase 1c: Convert anyOf → oneOf.
 *
 * oneOf is strictly more restrictive than anyOf (exclusive vs. inclusive OR).
 * For type-union patterns — the dominant anyOf use case in tool schemas,
 * where branches are mutually exclusive by type — the two are equivalent.
 * For the model generation use case (not validation), both keywords carry
 * identical guidance: "produce a value conforming to one of these schemas."
 *
 * oneOf is natively supported by Antigravity; anyOf is not.
 *
 * If a schema node has both anyOf and oneOf (extremely rare), anyOf is
 * dropped to avoid conflict — Phase 2 whitelist strips the residual.
 */
function convertAnyOfToOneOf(schema) {
    if (!schema || typeof schema !== 'object') return schema;
    if (Array.isArray(schema)) return schema.map(convertAnyOfToOneOf);

    const result = {};
    for (const [key, value] of Object.entries(schema)) {
        if (key === 'anyOf') {
            if (!schema.oneOf) {
                result.oneOf = Array.isArray(value)
                    ? value.map(convertAnyOfToOneOf)
                    : value;
            }
            // Both anyOf and oneOf present: drop anyOf, preserve original oneOf
        } else {
            result[key] =
                typeof value === 'object' && value !== null
                    ? convertAnyOfToOneOf(value)
                    : value;
        }
    }
    return result;
}
```

**Edge case handling:**

| Scenario | Behavior |
|---|---|
| `anyOf` only | Renamed to `oneOf`; sub-schemas recursively processed |
| Both `anyOf` and `oneOf` present | `anyOf` dropped; original `oneOf` preserved |
| No `anyOf` in schema | Passes through unchanged |

#### 4.2.2 Keyword Registry Updates

**`ALLOWED_KEYWORDS`** — remove `'anyOf'`:

```

**`KNOWN_UNSUPPORTED_KEYWORDS`** — add `'anyOf'`:

```

**`MONITORABLE_KEYWORDS`** — no change. `anyOf` is actively transformed (not stripped), so monitoring its occurrence adds no actionable information.

#### 4.2.3 Updated Orchestration

```javascript
function sanitizeSchemaForAntigravity(schema, logger) {
    if (!schema || typeof schema !== 'object') return schema;

    let result = schema;

    // ── Phase 1: Semantic Transforms ──
    result = convertConstToEnum(result);       // 1a: const → enum
    result = convertExclusiveBounds(result);   // 1b: exclusiveMin/Max → min/max + hint
    result = convertAnyOfToOneOf(result);      // 1c: anyOf → oneOf

    // ── Phase 2: Whitelist Filter ──
    result = filterToAllowedKeywords(result, logger);

    return result;
}
```

#### 4.2.4 Phase 2 Dead Code Cleanup

After removing `'anyOf'` from `ALLOWED_KEYWORDS`, the `key === 'anyOf'` branch in Phase 2's recursion logic becomes unreachable:

```javascript
// Before:
(key === 'allOf' || key === 'anyOf' || key === 'oneOf' || key === 'prefixItems')

// After:
(key === 'allOf' || key === 'oneOf' || key === 'prefixItems')
```

### 4.3 Design Rationale

**1. Why `anyOf → oneOf` instead of description hints — the core design decision**

Three approaches were evaluated for handling `anyOf` removal:

| Approach | Type structure | Description impact | Complexity | Information fidelity |
|---|---|---|---|---|
| **A: Description hint** | Lost — property becomes typeless `{}` | Bloat risk | Medium | Low (free-text) |
| **B: Type hoisting** (pick first branch) | Partial — one branch only | Minimal | Medium-High | Medium |
| **C: `anyOf → oneOf` conversion** | Fully preserved | None | Low | High — strictly more restrictive; exact for type unions |

**Approach A (Description hint) — rejected.** `anyOf` is a structural Applicator keyword, not metadata. Stripping `anyOf` and adding a description like `"(anyOf: string | number)"` leaves the property with no `type` field and no sub-schema structure. The model receives an effectively **empty schema** with a free-text hint:

```json
// Approach A result — property loses ALL type information:
{ "contact": { "description": "(anyOf: string (email) | number)" } }

// Approach C result — full structure preserved:
{ "contact": { "oneOf": [{ "type": "string", "format": "email" }, { "type": "number" }] } }
```

Approach C preserves full type structure, sub-schema constraints (`format: "email"`), and requires no free-text parsing by the model. Description hints are appropriate for **metadata** keywords (like `exclusiveMinimum`, which has a close numeric equivalent with `minimum`) but not for **structural** keywords that define the valid data shape.

Additionally, `anyOf` sub-schemas can be arbitrarily complex (nested objects, arrays, multiple constraints). Serializing them into a description string would either be too verbose (causing description bloat — already identified as a risk in RFC-006 §7) or too lossy (dropping structural detail that a proper schema representation preserves).

**Approach B (Type hoisting) — rejected.** Selecting the "best" branch and promoting its type introduces heuristic complexity (which branch is "best"?), discards all non-primary branch information, and is unnecessary when a structurally complete alternative exists.

**Approach C (`anyOf → oneOf`) — selected.** The conversion is:

- **Structurally lossless** — all sub-schema branches, types, constraints, and hierarchy preserved (the keyword is renamed; no sub-schema content is altered or discarded)
- **Semantically conservative** — `oneOf` is a strictly more restrictive subset of `anyOf` (exclusive vs. inclusive OR). Values matching multiple branches are rejected under `oneOf` but accepted under `anyOf`. This tightening is acceptable because (a) type-union branches are mutually exclusive by `type`, and (b) for model generation the distinction is irrelevant (§4.3 rationale #2)
- **Natively supported** — `oneOf` is in `ALLOWED_KEYWORDS` and confirmed by production
- **Implementation-trivial** — single recursive key rename, identical pattern to the existing `const → enum` transform (Phase 1a)
- **Zero description impact** — no field bloat, no free-text reliability concerns

**2. `anyOf` and `oneOf` are NOT semantically equivalent — but the difference does not matter here**

`anyOf` and `oneOf` have distinct formal semantics:

| Keyword | JSON Schema definition | Set-theoretic model |
|---|---|---|
| `anyOf: [A, B]` | Value matches **at least one** of A, B | A ∪ B |
| `oneOf: [A, B]` | Value matches **exactly one** of A, B | (A ∖ B) ∪ (B ∖ A) = A △ B |

`oneOf` is a **strictly more restrictive** subset of `anyOf`: every value accepted by `oneOf` is also accepted by `anyOf`, but not vice versa. The difference manifests when a value matches **multiple** branches simultaneously:

```
anyOf: [A, B]  →  accepts values in A∩B (matching both)
oneOf: [A, B]  →  rejects values in A∩B (matching both)
```

**Concrete example of divergence:**

```json
{
  "anyOf": [
    { "type": "object", "properties": { "name": { "type": "string" } }, "required": ["name"] },
    { "type": "object", "properties": { "age": { "type": "integer" } }, "required": ["age"] }
  ]
}
```

Input `{ "name": "Alice", "age": 30 }` matches both branches — valid under `anyOf`, **invalid** under `oneOf`.

**Why this divergence does not affect tool schema sanitization:**

| Factor | Analysis |
|---|---|
| **Dominant `anyOf` pattern** | Type unions (`anyOf: [{ type: "string" }, { type: "number" }]`) — branches are mutually exclusive by `type`. A value can only match one branch. `anyOf` ≡ `oneOf` for this pattern. |
| **Generation vs. validation** | The schema appears in `functionDeclarations` to **guide model generation**, not to validate user input. For generation, the model selects one branch and produces a conforming value — identical behavior regardless of `anyOf`/`oneOf`. A model does not intentionally generate values matching multiple branches. |
| **Restrictiveness direction** | `oneOf` is strictly **more restrictive** than `anyOf`. If Antigravity performs post-hoc validation, `oneOf` may reject some values that `anyOf` would accept (false negatives), but will never accept values that `anyOf` would reject (no false positives). This is the conservative/safe direction. |
| **Overlapping-branch rarity** | In tool schemas, `anyOf` with genuinely overlapping branches (where A∩B ≠ ∅ and a valid value could land in the intersection) is rare. The common patterns — type unions, nullable types (`Optional[T]`) — have disjoint branches by construction. |

**Summary:** The conversion is **not semantically equivalent in the general case** — it is a conservative approximation that tightens the constraint from inclusive OR to exclusive OR. For the specific context of tool schema generation guidance with type-union patterns, the tightening has no observable effect on model behavior.

**3. Consistency with RFC-006 Phase 1 design pattern**

The `anyOf → oneOf` transform follows the same structural pattern as existing Phase 1 transforms:

| Transform | Unsupported keyword | Supported equivalent | Semantic fidelity |
|---|---|---|---|
| Phase 1a | `const` | `enum` | Exact (logical equivalence) |
| Phase 1b | `exclusiveMinimum` | `minimum` + description hint | Exact for integers; approximate for numbers |
| **Phase 1c** | **`anyOf`** | **`oneOf`** | **Exact for type unions (disjoint branches); conservative tightening for overlapping branches** |

All three follow the pattern: unsupported keyword → natively supported equivalent. Phase 1c is the only transform that introduces a **stricter** constraint (exclusive OR vs. inclusive OR) rather than a looser one — the trade-off analysis is in §4.3 rationale #2.

**4. Why not the monitor-first approach for `anyOf`**

Unlike most keywords in `MONITORABLE_KEYWORDS` (which are infrequent in tool schemas and have uncertain transform paths), `anyOf` has:

- **Confirmed production occurrence** — documented in `odin/logs/body.json`
- **A clean supported equivalent** — `oneOf` is structurally identical
- **High expected frequency** — Python `Optional[T]` generates `anyOf: [T, null]`, affecting many MCP tools

The monitor-first approach (strip, track frequency, implement transform later) is appropriate for keywords with uncertain usage and complex transform paths. `anyOf` has neither uncertainty: it appears in production today, and `oneOf` is its ready-made equivalent.

## 6. Testing Strategy

The testing approach covers: (1) transform correctness for all `anyOf` patterns, (2) interaction with existing Phase 1/Phase 2 transforms, and (3) end-to-end Antigravity acceptance.

| # | Scenario | Input | Expected |
|---|----------|-------|----------|
| 1 | Simple type union | `{ anyOf: [{ type: "string" }, { type: "number" }] }` | `{ oneOf: [{ type: "string" }, { type: "number" }] }` |
| 2 | Nullable pattern | `{ anyOf: [{ type: "string" }, { type: "null" }] }` | `{ oneOf: [{ type: "string" }, { type: "null" }] }` |
| 3 | anyOf with sub-schema constraints | `{ anyOf: [{ type: "string", format: "email" }, { type: "number" }] }` | `{ oneOf: [{ type: "string", format: "email" }, { type: "number" }] }` |
| 4 | Nested anyOf in property | `{ properties: { x: { anyOf: [{ type: "string" }, { type: "integer" }] } } }` | `{ properties: { x: { oneOf: [{ type: "string" }, { type: "integer" }] } } }` |
| 5 | anyOf within anyOf branches | `{ anyOf: [{ anyOf: [{ type: "string" }, { type: "null" }] }, { type: "number" }] }` | `{ oneOf: [{ oneOf: [{ type: "string" }, { type: "null" }] }, { type: "number" }] }` |
| 6 | Both anyOf and oneOf present | `{ anyOf: [{ type: "string" }], oneOf: [{ type: "number" }] }` | `{ oneOf: [{ type: "number" }] }` — original `oneOf` preserved, `anyOf` dropped |
| 7 | No anyOf in schema | `{ type: "string", enum: ["a", "b"] }` | Unchanged |
| 8 | Production payload (`body.json`) | Full tool schema with `anyOf` contact field | `anyOf` → `oneOf`; all other fields preserved; Antigravity accepts |
| 9 | anyOf + Phase 2 interaction | `{ anyOf: [{ type: "string", title: "Name" }] }` | `{ oneOf: [{ type: "string" }] }` — Phase 1c converts anyOf, Phase 2 strips `title` |
| 10 | Phase 1a + 1c combined | `{ anyOf: [{ const: "active" }, { type: "number" }] }` | `{ oneOf: [{ enum: ["active"] }, { type: "number" }] }` — `const → enum` + `anyOf → oneOf` |

## 7. Risks

| Risk | Likelihood | Impact | Mitigation |
|------|-----------|--------|------------|
| `oneOf` also not supported by Antigravity | Low | High — same 400 errors | Test with production payload before deployment; if confirmed, fall back to stripping with description hint |
| Antigravity performs strict `oneOf` post-hoc validation on model output | Low | Med — model-generated values matching multiple branches rejected as invalid, causing tool call failures that would not occur with `anyOf` | Monitor for unexpected tool call validation errors after deployment; if observed, implement branch-disjointness check or fall back to first-branch hoisting |
| `anyOf` with genuinely overlapping branches (A∩B ≠ ∅) degrades under `oneOf` | Low | Low-Med — model could generate a value in the intersection that passes `anyOf` but fails `oneOf` validation | For generation, the model selects one branch (no multi-branch targeting). Risk exists only if post-hoc validation is strict. Add warn-level logging for `anyOf` schemas where branches share the same `type` (heuristic overlap indicator) |
| `allOf` is also not supported (same false assumption as `anyOf`) | Low-Med | High — different class of 400 errors | Out of scope for this RFC; proactive verification recommended in Future Work |
| Schemas with both `anyOf` and `oneOf` lose `anyOf` information | Very Low | Low — `oneOf` provides sufficient type guidance | Edge case; original `oneOf` preserved; `anyOf` content accessible via request logs if debugging needed |

## 8. Future Work

- **Proactive `allOf` / `oneOf` support verification** — RFC-006 assumed all three Applicator composition keywords are natively supported. `anyOf` has been disproven. `allOf` and `oneOf` should be proactively tested against Antigravity to prevent the same class of production failure.
- **Nullable pattern optimization** — `anyOf: [{ type: T }, { type: "null" }]` (from Python `Optional[T]`) could be converted to `{ type: T, nullable: true }` if Antigravity supports the `nullable` keyword, producing a simpler schema than `oneOf` for this common pattern.
- **`anyOf`/`oneOf` validation divergence monitoring** — if Antigravity validates model output against `oneOf` strictly (rejecting values matching multiple branches), monitor for generation quality regressions.
- **RFC-006 errata** — update RFC-006 §4.3 rationale #8 and §4.2.1 `ALLOWED_KEYWORDS` documentation to reflect the corrected `anyOf` status.
