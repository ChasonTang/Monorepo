## Review Discipline

- Report only issues affecting requirement satisfaction, design correctness, testability, executability, or implementation risk. Ground each in the requirement, template, RFC text, or verifiable project facts.
- Do not invent issues to look thorough â€” a passing review is valid. Do not flag writing preferences, unrelated refactors, or requirements outside the template or original requirement.
- A single-RFC hard-cap violation is a `Blocker`, not a `Major`. The recommendation is split/re-scope, not "trim wording" or "continue patching this RFC", unless a valid explicit exception is present.
- `first` review is the exhaustive pass: surface every qualifying issue, descending severity, without stopping at the first Blocker.
- `re-review`: uphold a `Rejected` / `Partially accepted` finding only when its Basis is unsound or contradicted by the revised RFC; reopen an `Accepted` finding only when the revision did not address it. Raise a `New` finding only against text this revision changed or a regression it introduced â€” never re-mine sections the `first` review already cleared.
- **Grep-sweep contradictions.** When a finding identifies a factual error or contradiction (a claim conflicting with cited code, another RFC, or another section), Evidence must list *every* occurrence in the RFC. Pointing at one instance of a multi-instance contradiction forces another round to clean up the rest.
- **Revision response form.** A `re-review` revision response missing the required `Self-Check Evidence` block (see `rfc_revision_prompt.md` Output Format item 3), or whose Evidence sub-bullets assert "I verified X" without pasting the verifying command + output, is treated as if no self-check was run. Scrutinize every revision-changed claim for stale identifiers, numeric inconsistencies, mechanism side effects, and missed cross-section ripple. Any regression introduced by the revision is `New` with no Origin protection. The form defect itself is `Minor` (does not gate Pass on its own) but escalates every regression it lets through.
- **Patch fatigue escalation.** When a `Blocker` / `Major` finding has been `Reopened` â‰Ą2 prior cycles on the same code site (same Â§3.2 mechanism, same struct field, same lifecycle hazard, same trust boundary), the Recommendation must escalate from "another patch" to "redesign the contract at this site" â€” and cite the prior Change attempts that proved insufficient. Three failed patches on one mechanism is a signal that the contract surface (not the mechanism's implementation) is what needs to change. Continuing to recommend the next variant of the same patch shape is how a six-round review converges through compounding band-aids instead of converging through one redesign.
- `Minor` findings never gate the verdict; do not manufacture them to force another round.
