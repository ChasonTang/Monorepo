# RFC-011: Odin Bidirectional Byte Relay

## 1. Summary

Add `odin/relay.{c,h}`, a single-thread, event-loop-driven bidirectional byte relay: given two caller-owned nonblocking file descriptors and an `odin_event_loop_t`, `odin_relay_start` watches both fds through the RFC-010 `odin_event_io_*` API and forwards bytes in each direction through its own fixed 64 KiB backpressure buffer (stop reading a source when its destination buffer is full, resume when the destination drains), propagates end-of-stream as a half-close (`shutdown(SHUT_WR)` on the peer fd after flushing, with the opposite direction still flowing), classifies an `ODIN_EVENT_ERROR` readiness as a graceful end-of-stream or a genuine fault by draining the source, aggregates a genuine `read`/`write`/`shutdown` error into one teardown, and invokes a caller completion callback exactly once with `ODIN_RELAY_OK` or `ODIN_RELAY_ERROR` plus the failing `errno` — never closing, DNS-resolving, or selecting the transport for the fds.

## 2. Goals

- **G1.** Given two caller-owned nonblocking fds and a loop, deliver every byte each fd's peer sends to the opposite fd, in order, in both directions concurrently.
- **G2.** Forward a stream larger than the buffer using bounded memory — at most 64 KiB buffered per direction — without dropping or reordering bytes when one side drains slower than the other fills.
- **G3.** When one fd reaches end-of-stream, deliver the bytes already buffered toward its peer and then half-close that peer's write side via `shutdown(SHUT_WR)`, while the opposite direction keeps forwarding until it too reaches end-of-stream.
- **G4.** Invoke the caller's completion callback exactly once — `ODIN_RELAY_OK` with `err == 0` when both directions reached end-of-stream and were half-closed, or `ODIN_RELAY_ERROR` with the failing `errno` when a genuine `read`/`write`/`shutdown` (or asynchronous socket) error tears the relay down; a peer's graceful close, although the loop reports it as `ODIN_EVENT_ERROR`, completes a direction rather than tearing the relay down.
- **G5.** Never close, DNS-resolve, or select the transport for the caller's fds; the caller owns the relay object and frees it with `odin_relay_destroy` (callable from within the completion callback), and neither completion nor `destroy` closes the fds.

## 3. Design

### 3.1 Overview

`odin/relay` is a new leaf module in the Odin library and the first in-tree consumer of the RFC-010 event-loop I/O-watch API (`odin_event_io_start` / `_update` / `_stop`). The caller constructs an `odin_event_loop_t`, opens and connects two nonblocking fds (for the proxy: the browser-facing socket and the upstream-facing socket), then calls `odin_relay_start(loop, fd_a, fd_b, on_done, user_data, &relay)`. The relay registers one I/O watch per fd on that loop and from then on runs entirely from the loop's readiness callbacks on the loop's owner thread; it starts no thread, opens no fd, runs no timer, and performs no name resolution or transport negotiation. Data enters by `read(2)` on whichever fd the loop reports readable and leaves by `write(2)` on the opposite fd; between the two it sits in one of two fixed-capacity per-direction buffers. Control returns to the caller only through `on_done`, which fires once when both directions have closed or when the relay tears down on error. The two fds and the relay object are caller-owned: the relay neither closes the fds nor frees itself. Because it touches only two ready-driven byte streams, the relay is independent of whether the upstream transport is TCP or QUIC.

```
   peer A            relay (odin_relay_t)            peer B
    fd_a  --read-->   [ buffer A->B, 64 KiB ]  --write-->  fd_b
    fd_a  <-write--   [ buffer B->A, 64 KiB ]  <--read---  fd_b
                  watches fd_a / fd_b via odin_event_io_*
                  on the caller's loop; on completion or
                  error -> on_done(relay, status, err, user_data)
```

### 3.2 Detailed Design

#### 3.2.1 Public API and Lifecycle

```c
/* odin/relay.h */
#include "odin/event_loop.h"

#ifdef __cplusplus
extern "C" {
#endif

typedef struct odin_relay_t odin_relay_t;

typedef enum odin_relay_status_t {
  ODIN_RELAY_OK = 0,
  ODIN_RELAY_ERROR,
} odin_relay_status_t;

typedef void (*odin_relay_done_cb)(odin_relay_t *relay,
                                   odin_relay_status_t status, int err,
                                   void *user_data);

int odin_relay_start(odin_event_loop_t *loop, int fd_a, int fd_b,
                     odin_relay_done_cb on_done, void *user_data,
                     odin_relay_t **out);
void odin_relay_destroy(odin_relay_t *relay);

#ifdef __cplusplus
}
#endif
```

**Unstated contract.**

- **Ownership & preconditions.** The caller owns both fds and the relay object; the relay closes neither fd and frees only its own state. `fd_a` and `fd_b` must be distinct, open, nonblocking, caller-owned **connected stream sockets** — the §3.2.3 half-close and error paths call `shutdown(SHUT_WR)` and `getsockopt(SO_ERROR)`, which are `ENOTSOCK`/undefined on pipes or regular files — and `on_done` must be non-null. `odin_relay_start` is an owner-thread API under the RFC-010 contract — callable before `run`, from a loop callback, or from a posted task.
- **Success.** Registers one READ-only watch per fd — `fd_a`'s watch first, then `fd_b`'s (both buffers start empty, both read sides open, nothing to write) — writes `*out`, and returns `0`.
- **Failure & rollback.** Returns `-1` with `errno` set, writes nothing to `*out`, and leaves no watch registered: `ENOMEM` when the relay, a buffer, or an I/O handle cannot be allocated, otherwise the `errno` from `odin_event_io_start` (including `EEXIST` when either fd already has an active watch in this loop — `fd_a != fd_b` is a precondition above, not a recoverable input, so duplicate-fd is undefined behavior the relay does not promise to diagnose); if the first (`fd_a`) watch was registered before the second (`fd_b`) failed, `start` stops the first and frees partial state before returning.
- **Completion (`on_done`).** Fires exactly once on the owner thread, as the relay's **final action** during teardown: `ODIN_RELAY_OK` with `err == 0` when both directions reached end-of-stream, or `ODIN_RELAY_ERROR` with `err` set to the best-effort failing `errno` (nonzero on every error teardown — see §3.2.4; a bare hangup whose `SO_ERROR` is already clear is absorbed as EOF, not a fault — see §3.2.3) — `status` is the authoritative signal. The relay accesses no relay state after `on_done` returns, so `odin_relay_destroy(relay)` is legal from inside `on_done`.
- **Destroy.** `odin_relay_destroy` stops any still-active watches (their handle memory is reclaimed by the loop's deferred-free per RFC-010 §3.2.1), frees the two buffers and the relay object, never closes `fd_a`/`fd_b`, and never invokes `on_done`; `odin_relay_destroy(NULL)` is a no-op, and the pointer is dead afterward. `destroy` is how a caller both reclaims a completed relay and aborts a still-running one.
- **Threading.** All entry points and internal callbacks are owner-thread; the relay adds no locks.

Satisfies: G4 via the two-value status enum and the exactly-once, final-action `on_done` surface; G5 via caller-owned fds and object, the close-free-nothing `start`/`destroy` boundary, `start`'s failure rollback that frees partial state and leaves no watch registered, and `destroy` being callable from within `on_done`.

#### 3.2.2 Readiness Reconciliation and Backpressure

```c
/* Per-fd desired event mask, recomputed after every read / write / EOF.   */
/* Each fd SOURCES one direction (read fills its buffer) and SINKS the      */
/* other (write drains its buffer). CAP = 65536 (64 KiB) per direction.     */
want_read(fd)  = src(fd).read_eof == 0 && src(fd).len <  CAP
want_write(fd) = sink(fd).len > 0
mask(fd)       = (want_read(fd)  ? ODIN_EVENT_READ  : 0)
               | (want_write(fd) ? ODIN_EVENT_WRITE : 0)
/* mask(fd) == 0  ->  stop the watch (the loop forbids an empty mask).      */
```

**Unstated contract.** Dispatch is level-triggered (RFC-010 §3.2.2): the relay issues at most one `read` and one `write` per readiness callback for a given fd and relies on the loop re-reporting a still-ready fd, so no fd starves and no inner drain loop is needed. `do_read` reads into the contiguous free run at `tail` — `min(CAP - len, buf_end - tail)`, which is `> 0` whenever READ is watched — so a direction never buffers more than `CAP` bytes; the moment `len == CAP`, `want_read` is false and the READ bit leaves the mask (this is the backpressure stop), and a later `write` that frees space restores it. `EAGAIN`/`EWOULDBLOCK`/`EINTR` from `read`/`write` mean "no progress, wait for the next readiness" and return without error; any other `errno` triggers teardown (§3.2.4). Because the loop forbids an empty mask, a fd that wants neither READ nor WRITE has its watch stopped and re-`start`ed (a fresh handle) when it next wants one — `mask(fd) == 0` arises whenever a fd's read side is unreadable (`src(fd).read_eof` set **or** `src(fd).len == CAP`) while its sink buffer is empty (`sink(fd).len == 0`), so it covers both a half-closed fd whose surviving write buffer drains to empty and later refills **and** a still-open fd (`read_eof == 0`) under one-way backpressure whose source buffer has filled to `CAP` while the reverse direction is momentarily idle, and a failed re-`start` becomes a teardown.

**Mechanism.**

```
do_read(r, d):                         /* d = src(fd); fills d's contiguous free run */
  run = min(CAP - d.len, d.buf_end - d.tail)   /* contiguous free bytes at tail, > 0 */
  n = read(d.src_fd, d.tail, run)
  if n > 0:    append n bytes to d;              return PROGRESS
  elif n == 0: d.read_eof = 1;                   return EOF      /* graceful end-of-stream */
  elif errno in {EAGAIN, EWOULDBLOCK, EINTR}:    return AGAIN
  else:        r.outcome = ERROR; r.err = errno; return FAILED

do_write(r, d):                        /* d = sink(fd); drains d's contiguous buffered run */
  run = min(d.len, d.buf_end - d.head)         /* contiguous buffered bytes at head, > 0 */
  n = write(d.sink_fd, d.head, run)
  if n > 0:    drop n bytes from d
  elif errno in {EAGAIN, EWOULDBLOCK, EINTR}: return
  else:        r.outcome = ERROR; r.err = errno

reconcile(r, fd):                      /* recompute and apply fd's watch mask */
  m = (src(fd).read_eof == 0 && src(fd).len < CAP ? ODIN_EVENT_READ : 0)
    | (sink(fd).len > 0 ? ODIN_EVENT_WRITE : 0)
  if m == r.cur[fd]: return
  if m == 0:             odin_event_io_stop(r.io[fd]); r.io[fd] = NULL
  elif r.io[fd] == NULL: if odin_event_io_start(r.loop, fd, m, on_ready, r,
                            &r.io[fd]) != 0: r.outcome = ERROR; r.err = errno; return
  else:                  if odin_event_io_update(r.io[fd], m) != 0:
                            r.outcome = ERROR; r.err = errno; return
  r.cur[fd] = m
```

The per-direction buffer is a fixed `CAP`-byte ring; `d.tail`/`d.head`/`d.buf_end` and the wrap arithmetic are implementation detail, with `do_read` targeting the contiguous free run at `tail` and `do_write` draining the contiguous buffered run at `head`. Each syscall's count is thus the contiguous run, never more than `CAP - d.len`, so the physical buffer end is never overrun; any wrapped remainder waits for the next level-triggered re-arm.

Satisfies: G1 via the read-into-buffer / write-from-buffer forwarding that preserves byte order per direction; G2 via the contiguous-run read bound (`≤ CAP - len`) and the `want_read`/`mask` rule that stops a source's READ watch at `len == CAP` and resumes it when a write frees space.

#### 3.2.3 Half-close and Completion

```c
/* Per direction d in {A = fd_a->fd_b, B = fd_b->fd_a}.                      */
d.read_eof   : a read() on the source fd returned 0 (graceful end-of-stream)
d.write_shut : shutdown(d.sink_fd, SHUT_WR) has been issued after d drained
/* ODIN_EVENT_ERROR is ambiguous -- RFC-010 ORs peer EOF/hangup AND hard     */
/* errors into it -- a hint to read, not a verdict; read() decides:          */
/*   read()==0 -> graceful EOF, half-close here;                             */
/*   read()==-1 / SO_ERROR != 0 (drained no bytes, no EOF) -> fault, §3.2.4. */
/* Half-close: d.read_eof && d.len == 0 && !d.write_shut                     */
/*               -> shutdown(d.sink_fd, SHUT_WR); d.write_shut = 1           */
/* Complete  : A.write_shut && B.write_shut  -> outcome = OK (on_done §3.2.4) */
```

**Unstated contract.**

- **Classification.** `ODIN_EVENT_ERROR` is **not** an unconditional teardown. RFC-010 delivers a peer's graceful `shutdown(SHUT_WR)`/`close` as `ODIN_EVENT_ERROR` (kqueue `EV_EOF` at `event_loop.c:941-942`; epoll `EPOLLRDHUP`/`EPOLLHUP` at `event_loop.c:896-901`, registered on every watch at `event_loop.c:434-435`) on a READ watch — the same output-only bit (`event_loop.h:43`) a hard error uses — so the relay cannot distinguish the two from the mask and must classify with `read()`, the common case for a CONNECT tunnel, not an exception. On `ODIN_EVENT_ERROR` (or `ODIN_EVENT_READ`) the callback drains the source: `read() == 0` is a graceful end-of-stream that sets `read_eof` and takes the drain → `shutdown(SHUT_WR)` half-close path below, while `read() == -1` with a real `errno` — or a nonzero `getsockopt(SO_ERROR)` when the drain yielded neither bytes nor EOF — is a genuine fault that tears the relay down (§3.2.4). An `ODIN_EVENT_ERROR` that is only a peer hangup (no readable data, `SO_ERROR == 0`) is absorbed as the EOF the read path already recorded, and the relay never re-reads a source once its `read_eof` is set.
- **Half-close and completion.** Within a callback the relay issues at most one `write` then one `read` for the ready fd and skips every remaining sub-step once `outcome != NONE`. A direction whose source hit EOF and whose buffer has fully drained gets exactly one `shutdown(SHUT_WR)` on its sink fd, propagating the end-of-stream; the opposite direction is untouched and keeps forwarding (half-close). A failed `shutdown` is a fault (§3.2.4). When both directions are `write_shut`, `drive` sets `outcome = OK` and the §3.2.4 teardown fires `on_done`.

**Mechanism.**

```
on_ready(loop, io, fd, events, ud):              /* ud = r (odin_relay_t *) */
  err = (events & ODIN_EVENT_ERROR) != 0          /* peer EOF/hangup OR hard error  */
  if r.outcome == NONE && sink(fd).len > 0 && ((events & ODIN_EVENT_WRITE) || err):
    do_write(r, sink(fd))                          /* flush queued bytes toward fd   */
  rd = AGAIN
  if r.outcome == NONE && src(fd).read_eof == 0 && src(fd).len < CAP
                       && ((events & ODIN_EVENT_READ) || err):
    rd = do_read(r, src(fd))                       /* PROGRESS | EOF | AGAIN | FAILED */
  if r.outcome == NONE && err && src(fd).read_eof == 0 && rd != PROGRESS:
    e = so_error(fd)                               /* getsockopt(SO_ERROR)           */
    if e != 0: r.outcome = ERROR; r.err = e        /* genuine async error -> teardown */
  if r.outcome == NONE: drive(r)
  if r.outcome != NONE: teardown(r)               /* §3.2.4; FINAL action, r untouched after */

drive(r):
  for d in {A, B}:
    if d.read_eof && d.len == 0 && !d.write_shut:
      if shutdown(d.sink_fd, SHUT_WR) != 0: r.outcome = ERROR; r.err = errno; return
      d.write_shut = 1
  reconcile(r, fd_a); reconcile(r, fd_b)          /* may set r.outcome on io failure */
  if r.outcome == NONE && A.write_shut && B.write_shut: r.outcome = OK
```

Satisfies: G1 via `drive`'s `d.len == 0` precondition that flushes a direction's buffered bytes before issuing its `shutdown(SHUT_WR)`, so the end-of-stream is delivered in order after the last byte; G3 via the per-direction `read_eof`/drain/`shutdown(SHUT_WR)` rule — reached through the `read() == 0` classification of an `ODIN_EVENT_ERROR` readiness — that half-closes one peer while the opposite direction keeps forwarding.

#### 3.2.4 Aggregated Teardown and Same-batch Safety

```c
/* outcome in {NONE, OK, ERROR}: set once, then teardown consumes it once.    */
/* ERROR is set by whichever fault fires first -- do_read/do_write (read or   */
/*   write fault, §3.2.2), drive (failed shutdown, §3.2.3), or on_ready       */
/*   (SO_ERROR != 0 on an ODIN_EVENT_ERROR that drained neither bytes nor     */
/*   EOF) -- carrying that call's errno.                                      */
/* Teardown : outcome == ERROR -> on_done(ERROR, err);                        */
/*            outcome == OK    -> on_done(OK, 0)  (dual write_shut, §3.2.3)    */
```

**Unstated contract.**

- **Aggregation.** Only a genuine fault tears down — a failed `read`/`write`/`shutdown`, or a nonzero `getsockopt(SO_ERROR)` when `ODIN_EVENT_ERROR` is set yet the drain yielded neither bytes nor EOF. Each such fault sets one `outcome = ERROR` plus the best-effort failing `errno` (nonzero on every error teardown), and `teardown` aggregates every error source — and the dual-`write_shut` `OK` — into one completion. `teardown` stops **both** I/O watches and then invokes `on_done` as its **final action**: no relay field is read or written after `on_done` returns (so `odin_relay_destroy` from inside `on_done` is safe), and a `torn_down` guard makes `teardown` idempotent.
- **Same-batch safety.** Same-batch use-after-free and double-completion — both watched fds ready in one backend batch while the first `teardown`, or an in-callback `odin_relay_destroy`, frees the relay — are barred by three defense-in-depth mechanisms layered over the loop's always-present active/generation check (RFC-010 §3.2.2, which drops any sibling whose watch was stopped before that entry is reached), no one of them alone sufficient across every path: (i) `teardown` stops **both** watches before returning, so the check drops a sibling already materialized in the same batch; (ii) the `torn_down` guard and `on_ready`'s `outcome` short-circuit (§3.2.3) make a sibling that still re-enters `on_ready` a no-op, so no second `on_done` fires; and (iii) `odin_relay_destroy` (§3.2.1) stops all still-active watches before freeing, covering the abort path where `teardown` never runs. §5 S2 and the §6 T8/T9/T10 rows verify these mechanisms across the with-`destroy` (T8), no-`destroy` (T9), and abort (T10) paths: T10 cleanly isolates (iii), while on the T8/T9 fault paths (i) drops the same-batch sibling and (ii) is the defense-in-depth backstop, so the two jointly guarantee a single `on_done`.

**Mechanism.**

```
teardown(r):
  if r.torn_down: return                          /* idempotent */
  r.torn_down = 1
  if r.io[fd_a]: odin_event_io_stop(r.io[fd_a]); r.io[fd_a] = NULL
  if r.io[fd_b]: odin_event_io_stop(r.io[fd_b]); r.io[fd_b] = NULL
  cb = r.on_done; ud = r.user_data
  st = (r.outcome == OK) ? ODIN_RELAY_OK : ODIN_RELAY_ERROR
  e  = (st == ODIN_RELAY_OK) ? 0 : r.err
  cb(r, st, e, ud)                                /* caller may odin_relay_destroy(r) here */
```

Satisfies: G4 via the single `outcome`-driven `teardown` that fires `on_done` exactly once — `OK` on dual end-of-stream completion, `ODIN_RELAY_ERROR` on a classified `read`/`write`/`shutdown` or `SO_ERROR` fault — behind the `torn_down` guard and the stop-both-watches step.

Not applicable — this RFC adds `odin/relay.{c,h}`, test-only `odin/relay_internal_test.h`, and `odin/relay_unittests.cpp` as brand-new files with no prior callers, and only appends them to existing GN targets, so nothing that compiled or ran before this RFC changes behavior.

## 5. Security

- **S1.**
  - **Threat:** Heap buffer overflow from a flooding peer. The relay moves attacker-controlled bytes between two sockets; a peer that sends one direction's data faster than the other end drains it (e.g., a multi-hundred-KiB burst while the consumer stalls) would, without a per-`read` bound, write past the end of the 64 KiB direction buffer and corrupt adjacent heap allocations. The trigger surface is either socket the relay watches — for the proxy, a remote HTTPS endpoint or a local browser, neither of which is trusted.
  - **Mitigation:** §3.2.2 bounds every `do_read` count argument to the contiguous free run `min(CAP - len, buf_end - tail)`, which is `≤ CAP - len` and never reaches past the physical buffer end on wrap, and the `want_read`/`mask` rule clears the READ bit (stopping the watch or dropping READ from the mask) as soon as `len == CAP`, so the loop stops delivering readable events for that fd until a `write` drains the buffer; total bytes buffered per direction never exceed `CAP = 65536`, and surplus bytes wait in the kernel socket buffer and the peer's send buffer.
  - **Enforcement:** §6 row T2 streams 256 KiB (far larger than `CAP` plus both socket buffers) through one direction with a consumer that lags, asserts the full payload arrives byte-exact and in order, and runs under the P2 AddressSanitizer command with no heap-overflow report.

- **S2.**
  - **Threat:** Use-after-free / double-completion from a stale same-batch event after teardown. Both watched fds can be ready within one materialized backend batch; a genuine error on one fd tears the relay down — and the caller may `odin_relay_destroy` it from inside `on_done` — after which the other fd's already-materialized entry could dispatch into freed or torn-down relay state, causing a use-after-free or a second `on_done`.
  - **Mitigation:** §3.2.4 pins the three defense-in-depth mechanisms that bar same-batch use-after-free and double-completion — (i) `teardown` stops **both** watches, (ii) the `torn_down` guard plus `on_ready`'s `outcome` short-circuit, and (iii) `odin_relay_destroy` (§3.2.1) stopping all still-active watches before freeing — each layered over the loop's always-present active/generation check (RFC-010 §3.2.2, `event_loop.c:731`) that drops any sibling whose watch was stopped before that entry is reached. No single mechanism covers the abort, in-callback-`destroy`, and no-`destroy` paths alone; the Enforcement below maps each path to its §6 row (T8/T9/T10) — T10 cleanly isolating (iii), and T8/T9 exercising (i) and (ii) jointly. `teardown` reads no relay state after `on_done` returns, so an in-callback `destroy` is safe.
  - **Enforcement:** §6 row T8 asserts the joint same-batch invariant by layering real RST sockets under RFC-010's queued synthetic backend batch hook: because that hook is consumed before the real backend wait (`event_loop.c:845`), the row first installs a `poll`-based arrival barrier on both relay fds (waiting for read/error/hangup, without `read` or `getsockopt(SO_ERROR)` so the pending RST is not consumed) so each relay `read` deterministically returns `ECONNRESET` rather than racing the RST and seeing `EAGAIN`; only then are both relay watch handles queued as `ODIN_EVENT_ERROR` in one `odin_event_loop_run` backend batch, and each corresponding relay `read` returns `ECONNRESET`. With `odin_relay_destroy` called from `on_done`, that deterministic same-batch batch yields exactly one `on_done` (`calls == 1`) and is ASan-clean, because the in-callback `destroy` frees the relay while the sibling's already-materialized entry is dropped at `event_loop.c:731` instead of re-entering the freed relay. Since that `destroy` stops the sibling watch whether or not `teardown` already did, T8 cannot exercise the no-`destroy` fault path, so two more rows cover the remaining paths: §6 row T9 uses the same queued double-error batch with **no** in-callback `destroy` to cover the no-`destroy` path, where teardown's stop-both-watches (i) and the `torn_down` guard (ii) jointly guarantee a single `on_done` — an implementation that **both** stops only the faulting watch **and** drops the `torn_down` guard re-enters the sibling and fires `on_done` twice (`calls == 2`), the compound failure T9 catches and T8 masks (in T8 the in-callback `destroy` stops the sibling via (iii), so `calls` stays `1` even under that mutant); §6 row T10 isolates (iii) — it aborts a still-running relay whose watches are genuinely active and then makes the fds readable, so a `destroy` that fails to stop active watches re-enters the freed relay under the P2 AddressSanitizer command. §6 row T4 exercises the no-relay-state-after-`on_done` half on a `write`-fault teardown.

## 6. Testing Strategy

**Shared fixture (fork deadline + watchdog).** All eleven rows call `odin_event_loop_run`, so each executes under the same `fork`/`waitpid` deadline fixture RFC-010 §6 established (extending the `WIFEXITED`/`WEXITSTATUS` harness at `odin/cli_unittests.cpp:380-392`), replicated in `relay_unittests.cpp`: the child runs the loop and every `ASSERT_*`/`EXPECT_*`, then `_exit(::testing::Test::HasFailure() ? 1 : 0)`, and the parent fails the row unless the child exits `0` within the 2 s deadline (`EventLoopRunDeadline`'s `kill(pid, SIGKILL)` + `FAIL` on overrun). Each row also arms a watchdog `odin_event_timer` (100 ms for the small-payload rows, 1 s for T2) that sets `state.timed_out` and calls `odin_event_loop_stop`; `on_done` records `status`/`err`/`calls` and stops the loop. For the assertion-red rows — T1 and T3–T11 — a relay that never reaches its expected outcome fails through the child's in-child `ASSERT_*`/`EXPECT_*` (the `timed_out`/`on_done`-count checks, T10's forwarding `read(pb)` check, or T11's partial-rollback `start` return) and the child's nonzero `_exit`, not the fork deadline. T3 is thread-driven yet still assertion-red: after `run` returns its body `close`s `fd_a`/`fd_b`, sending EOF to the reader thread's blocked `read(pb)` (and to the child's own `read(pa)`), so the child always `join`s and runs its `on_done.calls == 1`/`status` assertions — which fail against the no-forwarding stub. T2 alone surfaces red through the fork deadline, because its reader parks on a `written >= CAP` saturation gate (an atomic counter, not a blocking syscall) that no `close` can release; see the next paragraph.

**Disclosed hang-as-red deviation (T2).** T2 is the single row whose red signal is the fork deadline rather than an executed in-child assertion. Against the P1 no-forwarding stub nothing is forwarded, so T2's writer fills its pinned socket buffers and blocks, and its reader — which **defers its first read until the relay's ring saturates** (`written >= CAP`) — waits on a gate the stub never satisfies, because the unfilled ring never advances `written` to `CAP`; since the byte-exact tally consumes the joined writer/reader buffers, the child `join()`s before asserting, so after the watchdog stops the loop the child blocks in `join()` and surfaces red through the replicated 2 s fork deadline. This is an intentional, disclosed deviation from the template's preference that a red row "run the assertion and fail": against a *pure* no-forwarding stub no in-child assertion is reachable for T2, because the stub yields neither the forwarded bytes nor the ring saturation the gate awaits, so the child cannot advance from `join()` to its assertions without the very backpressure behavior under test. The hang is the deterministic red signal — with nothing forwarded the writer never unblocks past `CAP`+socket-buffer bytes and the gate never opens, so `join()`, and thus the child, can never complete within the 2 s deadline — and T1, which exercises the identical forwarding path single-threaded with assertion-based red, corroborates that the behavior the hang detects is forwarding. The other thread-driving row, T3, is **not** part of this deviation: its reader blocks in a plain `read(pb)`, so T3's post-`run` `close(fd_a)/close(fd_b)` sends the EOF that releases the reader and lets the child `join` and execute its `on_done.calls == 1`/`status` assertions against the stub (a run assertion that fails, not a timeout) — matching sibling RFC-010's practice (its P1 Done when) that the fork deadline catches a real P2 hang rather than flagging a red row. T2's saturation gate has no equivalent fd-close release (its reader parks on an atomic counter, not a blocking syscall), so its timeout-red is retained and the deviation is bounded to this one row.

**Per-row socket setup.** Rows T1–T6 and T11 build each fd pair as a `socketpair(AF_UNIX, SOCK_STREAM, 0)` and set `O_NONBLOCK` on only the relay's ends (`fd_a`, `fd_b`) — the §3.2.1 precondition — leaving the test-driven peer ends (`pa`, `pb`) blocking, so the threaded-driver rows (T2's writer/reader, T3's reader) drive a blocking peer fd exactly as `client_listen_unittests.cpp` T6 does; the relay receives one end of each pair (`fd_a`, `fd_b`) and the test drives the other (`pa`, `pb`), as RFC-009 §7 injects `socketpair` fds. T10 likewise uses idle `AF_UNIX` socketpairs but sets `O_NONBLOCK` on all four ends, since its single-threaded peer readback (`read(pb)`) must not block. T7 instead builds the relay's `fd_a`/`pa` as a loopback TCP connected pair (`AF_INET`/`SOCK_STREAM` over `127.0.0.1`, `fd_a` `O_NONBLOCK`, via the listen/connect/accept idiom of `client_listen_unittests.cpp` T9) and leaves `fd_b`/`pb` an idle AF_UNIX socketpair, and T8 and T9 build **both** the `fd_a`/`pa` and the `fd_b`/`pb` pairs as such loopback TCP connected pairs (`fd_a`/`fd_b` `O_NONBLOCK`), because on the `kqueue` backend these tests run on, an AF_UNIX socketpair peer-`close` surfaces only as a graceful EOF (`read() == 0`) — including under `SO_LINGER{l_onoff=1, l_linger=0}` or unread queued bytes — and so cannot provoke the `read() == -1`-with-real-`errno` fault these rows need, whereas a TCP peer `close`d after `SO_LINGER{l_onoff=1, l_linger=0}` sends an RST that makes the relay's `read` fail with `ECONNRESET`. For T8 and T9, three independent guarantees are needed and none subsumes the others. (a) **Deterministic `read` result:** the TCP RST (not an AF_UNIX EOF) makes the relay's `read` fail with `ECONNRESET`. (b) **Same-batch co-materialization:** `odin_event_loop_test_queue_backend_events` forces both dispatch entries into one backend batch, rather than assuming two loopback RSTs coalesce in one kqueue/epoll return. (c) **RST observability at dispatch time:** because the queued hook is consumed *before* the real backend wait (`event_loop.c:845`), it co-materializes the two entries without waiting for the loopback RST to actually reach the relay sockets — a synthetic dispatch fired right after `close` can outrun the RST, leaving the first relay `read` to see `EAGAIN` with `SO_ERROR == 0` and skip teardown. So after the abortive closes and the `odin_relay_test_io_handles` snapshot, both rows install an **arrival barrier**: they `poll` both relay fds (`fd_a`, `fd_b`) for `POLLIN | POLLERR | POLLHUP`, spinning past `EINTR` and partial readiness until *both* report ready, *without* calling `read` or `getsockopt(SO_ERROR)` so the pending RST/error the relay later observes is left intact. Only once both fds are poll-ready is each relay `read` guaranteed to return `-1`/`ECONNRESET` rather than `EAGAIN`; the queued batch then supplies the same-batch timing on top of that confirmed-observable RST. (This barrier was the gap in the prior revision, which forced the same batch but not RST observability at dispatch time.) The relay itself uses only the public `odin_event_io_*` API and only `read`/`write`/`shutdown`/`getsockopt` on the fds, so it is backend-agnostic; per the RFC-010 §6 constraint that Odin unit tests currently run only on macOS, these rows execute on the `kqueue` backend while the Linux `epoll` build is verified by cross-compilation.

**Test-only same-batch hook use.** `relay_unittests.cpp` compiles with `ODIN_RELAY_TESTING` and includes `odin/relay_internal_test.h`; the header declares `int odin_relay_test_io_handles(odin_relay_t *relay, odin_event_io_t **out_a, odin_event_io_t **out_b)`, which snapshots the relay's current `fd_a` and `fd_b` watch handles and returns `-1` with `errno == ENOENT` if either watch is inactive. T8 and T9 call this helper immediately after a successful `odin_relay_start`, then `poll` both relay fds to a ready state (read/error/hangup, without consuming the RST via `read`/`getsockopt` — the arrival barrier above), and only then pass the returned handles to RFC-010's `odin_event_loop_test_queue_backend_events` so the next `odin_event_loop_run` backend wait consumes one synthetic two-entry batch through the real run path and the same active/generation dispatch guard as kqueue/epoll.

Two of G4's four error-teardown sources are enforced by dedicated §6 rows — T7 the `read() == -1` clause (`ECONNRESET`) and T4 the `write` clause (`EPIPE`) of the shared `outcome = ERROR; err = errno` teardown. The other two — the failed-`shutdown(SHUT_WR)` clause of §3.2.3 `drive` and the `getsockopt(SO_ERROR)` fallback of §3.2.3 `on_ready` (its asynchronous-socket-error sub-case: `ODIN_EVENT_ERROR` set while the drain yields neither bytes nor EOF) — have no dedicated row, because neither has a deterministic trigger under the relay's already-connected stream-socket precondition (§3.2.1). The `shutdown(d.sink_fd, SHUT_WR)` in `drive` runs only on a sink fd that, by that precondition, is an open connected stream socket the relay never closes, so it fails only via `EBADF`/`ENOTSOCK`/`ENOTCONN` — none reachable on a connected fd whose direction just reached EOF — and otherwise succeeds even when the peer has already closed. The `SO_ERROR` fallback is likewise unreachable: a hard error on this backend is delivered through `read()`'s `errno` and consumed there, leaving `SO_ERROR == 0` afterward — the `ECONNRESET` T7 forces arrives exactly this way — and no deterministic trigger on a connected stream socket leaves `SO_ERROR` nonzero while `read()` returns neither bytes nor EOF. Both are therefore defensive teardown branches validated by inspection of §3.2.3 only, with no §6 row.

- **P1. Land the relay surface and red-verifiable `T1`–`T11` in the test binary only.**
